1.2 · Kritická pasáž

Krevní oběh n8n: JSON a Items

📄 Co je JSON

Není to kód — je to způsob, jak zapsat informace tak, aby jim rozuměl stroj i člověk.


Funguje na principu Klíč : Hodnota

{ "sku": "LN-X1C-2024", "nazev": "Lenovo ThinkPad X1", "kw_hlavni": "levny notebook", "pozice_gsc": 4 }

📦 Koncept Items (Položky)

Data v n8n neputují jako jeden text, ale jako „seznam krabic" — technicky: Array of JSON objects. Každá krabice = jeden Item.

📦 Item 1
{ThinkPad X1}
📦 Item 2
{levný notebook}
📦 Item 3
{/notebooky/}

= 1 produkt z feedu · 1 klíčové slovo z GSC · 1 URL ze sitemapy · 1 řádek z reportu

⭐ Zlaté pravidlo: Implicitní iterace

1️⃣
1 Item → Node se spustí 1×
1 produkt, 1 klíčové slovo, 1 URL ze sitemapy
🔟
50 000 Items → Node se spustí 50 000×
celý produktový katalog = 50 000 popisů na autopilotu
💡
Síla pro SEO specialistu:
Nemáš 1 produkt — máš 50 000 produktů. Každý je Item. AI napíše 50 000 SEO popisů automaticky, jeden po druhém — bez copy-paste.

2 · AI v n8n

Basic LLM Node vs. AI Agent Node

🧠

Basic LLM Node

Deterministický · Jednorázový

Pošle jeden prompt → dostane jednu odpověď. Konec. Žádná paměť, žádné nástroje, žádné rozhodování.

✅ Generování SEO popisů — vstup: parametry produktu → výstup: text
✅ Překlad meta tagů — vstup: CZ text → výstup: SK/EN verze
✅ Klasifikace intent — vstup: KW → výstup: info/transak/navi
✅ Extrakce parametrů — vstup: volný text → výstup: JSON se strukturou
💡
Basic LLM = výrobní pás. 50 000 produktů, každý dostane popis. Předvídatelné, levné, rychlé.
🤖

AI Agent Node

Autonomní · Vícekrokový

Agent sám rozhoduje, jaké nástroje použije a kolikrát — může volat vyhledávání, psát do DB, kontrolovat výsledky a opakovat.

✅ SEO audit kategorie — prohledá GSC, porovná s sitemap, napíše doporučení
✅ Konkurenční analýza — načte stránky konkurence, porovná, sumarizuje
✅ Chatbot s pamětí — pamatuje si kontext celé konverzace
✅ Výzkumný asistent — hledá, čte, ověřuje, iteruje sám do sebe

⚖️ Kdy použít co?

Basic LLM Vstup a výstup jsou jasné. Zpracování probíhá ve velkém a opakovaně.
AI Agent Úkol vyžaduje rozhodování, nástroje nebo víc kroků — výsledek předem neznáte.
3.1 · Komplexní automatizace

Mýtus nekonečné nudle

❌ Špatný přístup: Spaghetti workflow

Vizuální programování svádí k tomu stavět jedno obrovské workflow s desítkami uzlů a zcela generickými názvy.

⬡ HTTP Request 1 → ⬡ Set 2 → ⬡ IF 3 → ⬡ HTTP Request 4 → ⬡ Set 5 → ...
✗Těžko se hledají chyby
✗Kolegové se v tom nevyznají
✗Při úpravě jedné části se bojíte rozbít zbytek

✅ Cíl profíka

Stavět malá, specializovaná a opakovaně použitelná workflow. Každé dělá jednu věc a dělá ji dobře.

✓Každý modul testujete a opravujete samostatně
✓Oprava na jednom místě se projeví všude
✓Kolega může váš modul použít, aniž by mu rozuměl

🛠️ Dva nástroje, které nudle zkrotí

🔄 Loop Node Sekce 3.2

Krocení velkého objemu dat po dávkách — rate limiting, izolace chyb, kontrolovaná smyčka.

⚙️ Execute Workflow Sekce 3.3

Modulární architektura — volání podřízených procesů namísto jedné obrovské nudle.

📐 Pravidlo DRY

Don't Repeat Yourself — Neopakuj se.

Pokud stavíte stejnou sekvenci uzlů potřetí, je nejvyšší čas ji přesunout do samostatného sub-workflow a volat ji z každého workflow zvlášť.

💡
Složitost workflow není odrazem odbornosti — naopak. Jednoduchá a modulární workflow jsou znakem zkušeného automatizátora.
3.2 · Komplexní automatizace

Krocení velkého objemu dat: Loop

🛡 Obrana proti API limitům (Rate Limiting)

GSC API dovoluje max. 200 requestů za sekundu a 2 000 queries za den. Loop posílá dávky URL po 100 s Wait node 0,5 s pauzy — bez toho dostanete Error 429 při auditu 100 000 URL.

GSC API 200 req/s Semrush 10 req/s Wait node 0,5 s

🔒 Izolace chyb (Error Isolation)

Generujete 5 000 produktových popisů. U 312. selže AI API (timeout). Bez Loopu → celé workflow havaruje, ostatní popisy se nevygenerují. S Loopem → chybu zachytíte, přeskočíte a plynule pokračujete dál.

1. Vytvoř Child workflow

Nové workflow začínající uzlem Execute Workflow Trigger. Za něj poskládáte opakující se logiku (formátování zprávy, odesílání notifikace, výpočet ceny...).

2. Volej z Parent workflow

V hlavním procesu vložíte uzel Execute Workflow a v nastavení vyberete child ze seznamu. Tím ho „zavoláte" z libovolného počtu jiných workflow.

🔗 Volat lze ze 5, 10 i 50 různých Parent workflow najednou.

3. Předávání dat

Cokoliv do Execute Workflow nody vteče, přenese se do child jak je. Jakmile child skončí, výsledek se vrátí zpět a Parent pokračuje dál.

💡
Změníte formát SEO reportu v child workflow jednou → projeví se automaticky ve všech modulech (KW audit, sitemap diff, produktový report). Žádná ruční úprava na pěti místech.
3.4 · MacGyver sekce

Svatý grál: HTTP Request

💡 Tajemství nativních uzlů

Nativní nody pro Google Search Console, Semrush nebo Ahrefs v n8n pod kapotou nedělají nic jiného, než že posílají obyčejný HTTP Request. A tam, kde nativní noda chybí, stačí HTTP Request noda přímo — GSC API, Semrush API, Ahrefs, interní PIM.

🔌 4 věci, které vždy najít v API dokumentaci

1
URL / Endpoint
Kam posílám požadavek?
2
Metoda: GET vs. POST
GET = dej mi data · POST = tady máš data
3
Autentizace (Headers)
API klíč, Bearer Token...
4
JSON Body (u POST)
Data, která odesílám

📋 Příklad HTTP Request nastavení

Method POST
URL https://searchconsole.googleapis.com/v1/sites/{siteUrl}/searchAnalytics/query
Auth Bearer {{ $env.API_KEY }}
Body {"startDate": "{{ $json.date_from }}",
 "dimensions": ["query","page"],
 "rowLimit": 5000}

🏆 Klíčová výhoda

Jakmile ovládnete HTTP Request nodu, nepotřebujete čekat, až n8n přidá podporu pro vaši aplikaci. Napojíte se na cokoliv na světě, co má API.


Pro SEO: GSC Search Analytics API, Semrush Reports API, Ahrefs API, Screaming Frog API, interní PIM/CMS...

💡
Pokud HTTP node vyhodí chybu, přečtěte si JSON výstup — API vám obvykle přesně napíše, co jste zapomněli vyplnit.

3.6 · Nástroje profi automatizátora

Wait node a další hacky

⏱

Wait Node — časovač v průběhu

Pauza · Zdanit API · Webhooky

Wait node pozastaví běh workflow na nastavenou dobu — nebo počká na příchozí webhook odpověď z externí služby.

🔔
Rate limiting s Loop

Loop pošle 100 URL na GSC API → Wait 0,5 s → dalších 100. Bez toho: Error 429.

⌛
Odložené spouštění

Vygeneruj popis → počkej 24 h → pošli ho na kontrolu. Jedno workflow, žádný cron.

🔗
„Resume via webhook“ — čekání na schválení

AI vygeneruje popis → pošle link na schválení → Wait čeká, dokud šéf neklikne → pak teprve zapsat do CMS.

🚀 Další užitečné hacky

📈 Aggregate Node Sloučení

Sloučí všechny Items do jednoho (např. všechny popisy do jednoho e-mailu, nebo celý keyword list do jednoho CSV).

🔃 Merge Node Spojení toků

Spoj data ze dvou pipeline — např. GSC data + Semrush data po jednom klíčovém slovu (Inner Join).

📋 Set / Edit Fields Node Čistení dat

Přejmenování klíčů, odmazaní nepŕebytnych polí, doplnění výchozích hodnot — základní hygiena každého workflow.

🔧 n8n Environment Variables Bezpečnost

API klíče a tokeny uložte jako $env.GSC_API_KEY — nikdy nepsávat přímo do workflow.

4.1 · Reliability

Chyby jsou jistotou — Executions jsou záchrana

⚠️ Co se stane při chybě (výchozí)

Pokud node selže, celé workflow se okamžitě zastaví a zčervená. Další kroky se neprovedou.

🔘 Continue On Fail

Nachází se v nastavení každé nody (Settings).

Příklad: Workflow posílá task do Asany a e-mail. Asana má výpadek → bez Continue On Fail se zastaví i e-mail. S Continue On Fail → Asana chybu zaznamená, process jede dál.

🚨 Error Trigger (pro pokročilé)

Speciální záchranné workflow, které se automaticky spustí, když jakékoli jiné workflow spadne.

Typické použití: pošle SMS nebo Asana task — „Generator produktových popisů havaroval ve 3:14 u SKU LN-X1C-2024 — zkontrolujte, prosím."
Stačí vědět, že tato možnost existuje — až ji budete potřebovat, víte kde hledat.

🗂️ Executions: Letová data z černé skříňky

Levé menu → Executions = kompletní deník toho, co se skutečně stalo v produkci. Vše s přesnými daty a časovými razítky.

🔴
Debugging: Workflow spadla ve 3 ráno

Na prázdném plátně nic neuvidíte. Otevřete Executions, najděte červený běh → n8n ukáže přesný uzel, kde havaroval, včetně dat (Items), která do něj v tu chvíli tekla.

📋
Copy to Editor — Geniální funkce!

Zákazník napsal text místo čísla a databáze ho odmítla? Přišpendlete si vadný záznam z Executions zpět na testovací plátno — opravujete logiku bez toho, abyste formulář museli znovu ručně vyplňovat.

🟢
Úspěšné běhy jako dokumentace

Prohlédněte si i úspěšná provedení — vidíte přesně, jaká data prošla každým uzlem. Nejlepší způsob, jak pochopit, co vaše workflow opravdu dělá.

5 · Práce s daty

Google Sheets vs. PostgreSQL

Funkce 📗 Google Sheets 🐘 PostgreSQL
Rychlost nasazení ⚡ Blesková 🐢 Pomalejší
Spolehlivost při zátěži ❌ Nízká ✅ Extrémně vysoká
Lidská čitelnost ✅ Výborná ⚠️ Vyžaduje nástroj
Riziko rozbití ❌ Vysoké ✅ Minimální
Vyhledávání (Query) ⚠️ Pomalé & omezené ✅ Rychlé (SQL)

📗 Kdy zůstat u tabulek

  • Rychlé prototypování — ověřujete, jestli workflow vůbec funguje
  • Lidský faktor — výstup ráno zkontroluje obchoďák nebo účetní
  • Jednorázové exporty/importy — tabulka jako dočasný mezisklad

💥 Kde tabulky selhávají

Concurrency: 10 objednávek najednou → řádky se přepíší nebo API vrátí chybu
Error 429: Google má přísné API limity — rychle narazíte na strop zápisů
Křehkost: Kolega seřadí jediný sloupec → workflow se tiše a nenávratně rozbije

🐘 Kdy přejít na PostgreSQL

  • n8n pohání reálnou logiku — hlídáte stav (dostal uživatel uvítací e-mail?)
  • Velké objemy — desítky tisíc záznamů denně, SQL hledá v milisekundách
  • Striktní Schema — nikdo ji nerozbije nechtěným smazáním sloupce
🚀
Workflow evoluce: Začněte s Google Sheets → odlaďte logiku → až roste objem dat, vyměňte jen „koncovku" za Postgres nodu. Vizuální postup v n8n je skoro identický!

🔍 Automatický import KW dat

Stažujte data z několika zdrojů najednou a sloučte je pomocí Merge node do jednoho streamu Items — každé klíčové slovo = 1 Item.

🗂️ Intent clustering přes Switch node

Každé klíčové slovo = jeden Item. Switch node ho automaticky zařadí do košíku:

ℹ️ Informační „jak vybrat notebook“, „co je OLED“
🛒 Transakční „koupit notebook“, „cena levý laptop“
🗺️ Navigační „Lenovo ThinkPad Alza“, „HP notebooky“
⏰
Schedule Trigger: Každé pondělí stažni nová GSC data, porovnej pohyby pozic s minulým týdnem a pošli Asana report šéfce SEO týmu.

⚡ Clustering pipeline

1. 📥 GSC + Semrush → Items (každé KW = 1 Item)
↓
2. 🔧 Deduplikace + lowercasing (Code Node)
↓
3. 🧠 Switch node → intent klasifikace
↓
4. 📂 Přiřazení ke kategorii / cílové URL
↓
5. 💾 Uložení do KW databáze / Airtable

🗺️ Automatický sitemap monitoring

Každý týden stažuj aktuální sitemap.xml, porovnej s minulým během a detekuj změny:

✚ Nové URL — nové produkty nebo kategorie v indexu
✖ Odebrané URL — stažené produkty, smazé stránky
⚡ Přejmenované URL — riziko ztráty pozic bez 301 redirectu
📢
Asana task každé pondělí: „+127 URL přidáno · −43 odebráno · ⚠️ 12 přejmenovaných bez redirectu“

🔄 Sitemap diff + GSC = detekce kanibalizace

URL, které zmizely ze sitemap, ale stále se rankují v GSC → pravděpodobná kanibalizace nebo chybějící redirect.

Sitemap diff → missing URLs → GSC query → IF rank > 0 → ⚠️ Asana task
